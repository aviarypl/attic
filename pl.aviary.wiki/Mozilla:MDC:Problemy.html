Sytuacja w MDC - co dalej?

Mail Huberta nt. MDC: 

Witajcie,

Marek pytał ostatnio kilkakrotnie o MDC, obiecałem odpowiedź.

Z polskim MDC nie jest dobrze. Nowe wiki sprawia problemy, ale problemem
jest również sytuacja w której przez kilka miesięcy nie mogę się doprosić
odpowiedzi (http://bugs.aviary.pl/show_bug.cgi?id=2345 i
http://bugs.aviary.pl/show_bug.cgi?id=1935). Poprosiłem Ptaka, by zastanowił
się czy nie chce przekazać kierowania MDC komuś innemu. Jeśli zdecyduje się
to kontynuować - będzie musiał się dostosować do nowych wymagań. Jeśli nie
będzie chciał (bo np. ma dość męczenia się z nowym wiki, czy też nie ma
czasu) - poszukam nowej osoby na jego miejsce. Jeśli nie będzie chętnych -
MDC przestanie być naszym projektem. I może kiedyś do niego wrócimy, a może
i nie.

Bez względu na to jaką decyzję podejmie Ptak - zamierzam wprowadzić w życie
od 15 lipca następujące wymagania (za ich realizację będzie odpowiedzialny
szef polskiego MDC):
1. *Wszystkie* problemy związane z MDC muszą być zgłoszone w Bugzilli. Jeśli
problem jest po stronie Mozilli - bugzilla.mozilla.org, jeśli problem jest
po stronie Aviary.pl - bugs.aviary.pl.
2. Na stronie wiki projektu http://wiki.aviary.pl/Mozilla:MDC ma być lista
zgłoszonych bugów wraz z tytułami.
3. Co tydzień chciałbym widzieć wpis stand-upowy na wiki z podaniem co
zostało zrobione w ostatnim tygodniu: jakie bugi zgłoszone, jakie inne
działania podjęte. Poza tym jakie bugi zostały rozwiązane, jakie artykuły
były tłumaczone, poprawiane i plany na najbliższy tydzień.
4. Matrix w części dot. MDC musi podawać aktualny status raz na 3 miesiące

Po 3 miesiącach zrobię podsumowanie.

Jeśli okaże się, że ww. wymagania nie zostały spełnione, wtedy poszukam
nowego szefa MDC i tak do skutku. Jeśli nie będzie chętnych - MDC przestanie
być naszym projektem i przestaniemy udawać, że to tłumaczymy.

Jeśli okaże się, że ww. wymagania zostały spełnione, wtedy okres
3-miesięczny zostanie przedłużony o kolejne 3 miesiące i tak do skutku, aż
uznam, że sytuacja się unormowała.

MindTouch Deki bug priorities

This is a list of bugs I've filed with MindTouch for things I consider "must fix" issues to make Deki ideally suited to our needs, listed in order of urgency. 

http://bugs.developer.mindtouch.com/view.php?id=4875 Need to be able to make main page read-only without affecting new pages' permissions.  This may be evolving into a more traditional "lock this page" feature, which I love the idea of.

   
http://bugs.developer.mindtouch.com/view.php?id=5168 Need to be able to easily edit definition lists (dt/dd).  We use these extensively and presently they don't edit well at all in the WYSIWYG editor.
http://bugs.developer.mindtouch.com/view.php?id=4856  Searches are including languages other than the one you specify in the results.
http://bugs.developer.mindtouch.com/view.php?id=5956  The 'pagemain' skinning variable includes "class='disabled'" when it shouldn't. This results in the "View article" link to return from the talk page to an article being drawn as if it's disabled, even though it's not.
http://bugs.developer.mindtouch.com/view.php?id=6465 Make side-by-side diff view only show changes. Currently this view shows the entire article, making it hard to spot changes. This is tentatively scheduled to be fixed in the "Minneopa" release due to ship in late July.
http://bugs.developer.mindtouch.com/view.php?id=6414  Slow or non-responsive databases can leave the wiki non-functional even after the database is restored. This is tentatively scheduled to be addressed in the "Minneopa" release due to ship in late July.
http://bugs.developer.mindtouch.com/view.php?id=6368  Deki should support read-only database slaves. This would let us improve performance by load-balancing reads across multiple databases while keeping the databases in sync by only writing to one.
http://bugs.developer.mindtouch.com/view.php?id=5880  Add ability to include comments on file deletions. Being able to explain why files have been deleted (usually for policy violations) would be helpful.
http://bugs.developer.mindtouch.com/view.php?id=6492  Add "previous edit" and "next edit" links to diff pages. This would improve the ease of navigating while perusing through the history of an article.
http://bugs.developer.mindtouch.com/view.php?id=2579   Need to be able to preview articles before saving changes.
http://bugs.developer.mindtouch.com/view.php?id=6188  Add an option to predefine a set of tags and only allow those to be used. This is tentatively scheduled to be fixed in the "Minneopa" release due to ship in late July.
http://bugs.developer.mindtouch.com/view.php?id=4479 Need a "linked changes" tool.  We should be able to write something for this in DekiScript.
http://bugs.developer.mindtouch.com/view.php?id=4481  Add the ability to filter Popular Pages by language.
http://bugs.developer.mindtouch.com/view.php?id=4158  Add the ability to disable comments on a page-by-page basis.
http://bugs.developer.mindtouch.com/view.php?id=5829  Link editor window needs work. This is tentatively scheduled to be fixed in the "Minneopa" release due to ship in late July.
http://bugs.developer.mindtouch.com/view.php?id=6454  Add support for HTML 5 offline application based caching. This would improve performance by letting skin files, JavaScript, etc be cached locally on the user's system.
http://bugs.developer.mindtouch.com/view.php?id=4559  The "what links here" tool doesn't consider includes when used on template pages.
http://bugs.developer.mindtouch.com/view.php?id=6439  Instead of moving the editor toolbar around using JavaScript, use CSS to keep it locked in place. The bouncy editor toolbar is very disconcerting.
http://bugs.developer.mindtouch.com/view.php?id=5275 Add support for sorting tables. Having tables sorted makes them easier to use but currently we have to do this by hand. Tedious.
http://bugs.developer.mindtouch.com/view.php?id=4868  Need RSS feeds for recent changes in each language. These exist but are not exposed in the user interface anywhere.

Mozilla Developer Center - wersja polska

Tak jak już wcześniej obiecałem Gandalfowi, spróbuje opracować takie małe zestawienie dotyczące MDC PL (http://developer.mozilla.org/pl/). Na początek to może kilka statystyk z wiki MDC PL (http://developer.mozilla.org/pl/docs/Specjalna:Statistics):


Statystyki:

Grafika:S1.png

Stan na dzień: 8 stycznia 2007 r.

Kilka słów do powyższego zrzutu, otóż: pierwsza liczba to ogólna liczba stron jakie posiada polskie MDC. Kolejną z liczb na zrzucie jest liczba, która ku mojemu zdziwieniu nie podaje prawdziwego wyniku jaki powinien być wyświetlony. Czemu? Z moich obserwacji wynika, że przyczyną jest traktowanie przez system Wiki niektórych stron jako tych, które nie są artykułami (a w rzeczywistości zawierają treść). Zaliczane są do tego strony różnych dokumentacji, nie mające odpowiednio dużo treści (jest jakiś próg lecz nie za bardzo wiem jaki). Jak widać jest tych stron jest 1263 i w związku z tym co napisałem powyżej musimy dodać ok. 300 artykułów, co w przybliżeniu da nam ok 1563 artykułów na polskim MDC. Chciałbym przypomnieć, że w kwietniu 2006 na spotkaniu roboczym było coś ponad 500 artykułów. W liczbie tych artykułów są także arty do dotłumaczenia, których na tą chwilę jest 373, w tym ok 200 bardzo krótkich dotyczących dokumentacji XUL-a. Dosłownie po 2 lub 3 linijki. Reszta stron do dopracowania to kurs XUL-a lub końcówka dokumentacji JS i poradnika JS.

Jak widać MDC staje się coraz bardziej popularniejsza, można spotkać już wiele odnośników w polskim necie do MDC PL. Z moich obserwacji tej "marnej" statystyki wynika, że średnio ok 300 odwiedzin jest w ciągu godziny (w nocy), w granicach ok. 600 - 700 odwiedzin na godzinę w dzień (w południe). Można powiedzieć, że odwiedzalność stron MDC wzrasta bardzo szybko.

Kolejną liczbę, czyli liczbę edycji to wynik naszej działalności na MDC PL (w tym moich już około 9500), cała reszta edycji to wkład reszty, która juz pomogła na MDC.


Kto pomaga?

Otóż z tym jest różnie, na dzień dzisiejszy pomoc w tłumaczeniu uzyskuje od dwóch lub czasem trzech osób: Bedi (ogólnie wszystko tłumaczy), Witia (głównie CSS i HTML), Internauta1024(canvas). W przeszłości można było jeszcze spotkać na MDC te osoby: VooEak, Jan Dudek, PablO, Diablownik oraz kilkunastu innych tłumaczy, którzy bardzo szybko się wykruszyli. Pierwszym dwóm osobom jeśli była by taka możliwość chciałbym wysłać jakieś gadżety (o ile istnieje taka możliwość), w sumie Diablownika także można byłoby zaliczyć do osób więcej pomagających niż inni. Oczywiście nie wymieniam nikogo z Aviary.pl bo w sumie mniej, więcej wszyscy wiedzą ile włożyli swojego czasu i roboczogodzin. Dla porównania poniżej zamieszczam statystyki z MDC EN, tylko od razu dodam, że tam pracuje non stop kilka osób nad MDC, więc porównując ich do nas, proszę wziąć to pod uwagę:

Grafika:S2.png

I jak jest różnica? Widoczna od razu.

Co już mamy, a czego nam brak?

- mamy wiele artykułów, kilka ukończonych lub na ukończeniu ogromnych dokumentacji (XUL-a, JS), - brakuje nam przede wszystkim osób które by pomogły w tłumaczeniu, pomoc jest w 85 % opracowana - jeśli jest ktoś nowy to pomagam. - jeszcze raz napisze POMOCY, bo to, że teraz coś się dzieje - nie znaczy, że będzie się działo za miesiąc :P - jak już ustaliliście MDC nadaje się do promocji i pokazania - cieszy mnie to bardzo, z tego co wiem było wspominane, że dodajemy MDC do Wikipedii (z tym już sam zacząłem jakiś czas temu). Myślę, że warto by było zainteresować prasę IT, narobić szumu na grupach. Moim zdaniem przyniesie to korzyści przede wszystkim dla nas, bo zostaniemy zasileni nowymi tłumaczami (oczywiście niesie to za sobą też ujemne korzyści, ale do tej pory nie było zbyt wiele sytuacji kiedy robiłem revert).

- ok, nie będę się rozpisywał, bo nie wiem, co wiecie, a co nie. Proponuje jeśli są z czymś jakieś problemy lub pytania piszcie je i najwyżej odpowiem zbiorczo na wszystkie niejasności. Nie zamieszczałem pełnej listy tego co mamy bo bym ze dwie godziny wymieniał.
